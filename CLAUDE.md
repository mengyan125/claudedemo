# CLAUDE.md

本文件为 Claude Code (claude.ai/code) 在本仓库中工作时的指导规范。

## 一、沟通与文档规范

- **所有沟通一律使用中文**，包括代码注释、文档编写、commit message、PR 描述等
- **禁止随意修改前端代码**：修改前端代码前必须先与用户确认，获得明确同意后再执行
- **禁止随意生成文档**：不要主动生成操作文档、说明文档等，需要生成时先与用户确认
- **禁止提交 node_modules**：node_modules 不添加到源代码管理，不提交到 git
- compact 时务必保留：已修改文件列表、测试命令、当前阶段进度

## 二、项目概述

- **项目名称**：学生反馈系统
- **项目背景**：加强教学日常管理，学生的反馈是其中重要一个环节，需要给学生建立反馈的通道，让学生可以随时对教学、管理等方面提出意见和建议。
- **项目目标**：
- 建立学生与学校管理层之间的沟通渠道
- 保护学生隐私，支持匿名反馈
- 及时发现和解决教学管理中的问题
- 为学校管理决策提供数据支持

- **隐私保护**: 反馈信息只有特定管理员可查看，教师无法直接看到
- **及时响应**: 支持随时反馈，及时处理
- **权限分级**: 不同类别的反馈由不同管理员负责
- **数据安全**: 软件公司也无法查看反馈数据

## 三、技术栈

| 模块 | 技术栈 |
|------|--------|
| 前端 | Vue 3 + Vite + Webpack |
| 后端 | Spring Boot 2.x + JDK 1.8 + TiDB 8.5 + JetCache + Lettuce |

### 禁止使用

- jQuery、`var` 声明、`any` 类型（除非必要并注释原因）、未经确认的第三方库

## 四、架构与红线

### 架构分层

前端应用（页面 / 组件 / 状态）→ HTTP/WebSocket → API 层（路由 / 中间件 / 控制器）→ 数据层（模型 / 数据库）

### 不可违反的红线

1. 前后端必须通过 API 通信，禁止前端直接操作数据库
2. 所有 API 必须经过认证中间件（JWT 或 Session）
3. 敏感数据（密码、支付信息等）必须加密存储
4. 组件间通信必须通过状态管理（Pinia/Vuex），禁止直接 DOM 操作

### 通信协议

- RESTful API 规范
- 请求体字段：**camelCase**，URL 路径：**kebab-case**
- 统一响应格式：`{ "code": 200, "message": "success", "data": {} }`

## 六、编码规范

- 函数长度不超过 50 行，文件长度不超过 300 行（超过需拆分）
- 每个模块必须有 `index.ts` 导出
- 命名：组件 PascalCase，函数 camelCase，常量 UPPER_SNAKE_CASE

## 七、测试规范

- 所有新功能必须附带单元测试
- 接口测试覆盖率 > 95%
- 后端采用测试驱动开发（TDD），先写测试再实现功能
- 测试命令：`npm test`

## 八、Agent 协作规范

- **主 Agent 不写代码**：只负责任务统筹、分发、验收，避免上下文膨胀
- 调研和探索任务交给 Subagent 执行，要求返回**摘要**，禁止将大量原始代码带回主上下文
- 复杂任务组建 Agent 团队（前端、后端、数据库等），主 Agent 负责分解、协调与汇总
- 修改 DSL（接口定义、数据库 Schema）时必须通知所有相关 Agent
- 发现 bug 先通知相关 Agent 确认后再修复

## 九、上下文管理

- compact 后必须立即重读本文件（CLAUDE.md）
- 分阶段工作，每个阶段有明确的输入、目标和验收标准
- 上下文超过 180K tokens 时主动执行 `/compact`
- 约束文件读取范围，禁止无目的全局搜索
- 不相关任务之间执行 `/clear` 清空上下文
- 配合 `.claudeignore` 排除 `node_modules/`、`dist/`、`vendor/` 等噪音目录

## 十、数据库连接

- 默认仅使用 **MySQL**，禁止使用 H2/dev profile（除非用户在当次任务中明确要求）
- 启动后端时严格使用 MySQL 启动方式，不切换到 H2
- IP/端口/用户名/密码：待配置
- 连接需使用 proxy 模式：`socks://127.0.0.1:51080`
- 无法连接时与用户讨论解决方案，不要在本机安装数据库

## 十五、使用 Subagent 与 Agent 团队
- 需要调研代码、探索技术细节时**：调用子 Agent（subagent）执行，并要求其返回**摘要**，禁止将大量原始代码带回主上下文，避免污染主上下文。
- 组建 Agent 团队**：对于复杂任务，可组建由多个 Subagent 构成的团队，每个 Subagent 专注于特定领域（如前端、后端、数据库等）。主 Agent 负责任务分解、协调与汇总。团队内协作应遵循相同的摘要回传原则，避免冗余信息，并确保最终结果整合到主上下文中。
