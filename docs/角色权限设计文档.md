# 角色权限设计文档

## 一、权限模型概述

系统采用 **userType + role 双层权限模型**：

- **userType（用户类型）**：粗粒度分类，区分用户身份
  - `student` — 学生
  - `teacher` — 教师
  - `admin` — 管理员

- **role（角色）**：细粒度权限控制，针对管理功能的权限划分
  - `SYSTEM_ADMIN` — 系统管理员
  - `ROLE_ADMIN` — 角色管理员
  - `CATEGORY_ADMIN` — 类别管理员

**关键规则**：
- 学生只能访问学生端功能，无需分配角色
- 管理员通过 `sys_user_role` 表分配角色，角色决定可访问的管理功能
- **教师可兼任管理员**：教师的 userType 保持 `teacher` 不变，通过分配 admin 角色获得管理权限

## 二、角色权限矩阵

### 2.1 三种管理角色的权限边界

| 功能模块 | SYSTEM_ADMIN | ROLE_ADMIN | CATEGORY_ADMIN |

| **反馈统计**（统计图表/教师排名） | ✅ | ✅ | ❌ |
| **反馈管理**（类别增删改/启停） | ✅ | ✅ | ❌ |
| **快捷回复管理** | ✅ | ✅ | ✅ |
| **用户管理**（增删改/启停/批量创建） | ✅ | ❌ | ❌ |
| **学期管理** | ✅ | ❌ | ❌ |
| **年级管理** | ✅ | ❌ | ❌ |
| **班级管理** | ✅ | ❌ | ❌ |
| **师生关系管理** | ✅ | ❌ | ❌ |
| **角色权限设置**（分配/撤销角色） | ✅ | ❌ | ❌ |
| **类别权限配置**（分配类别管理员） | ✅ | ✅ | ❌ |

### 2.2 用户类型与功能对应

| 用户类型 | 可访问功能 |
|---------|-----------|
| student | 我要反馈、反馈跟踪（仅自己的反馈） |
| teacher（无admin角色） | 无可用功能页面（提示联系管理员分配权限） |
| teacher（有admin角色） | 根据分配的角色访问对应管理功能 |
| admin（有admin角色） | 根据分配的角色访问对应管理功能 |

## 三、数据库设计

### 3.1 相关表结构

```
sys_user            用户表（含 user_type 字段）
    ↓ 1:N
sys_user_role       用户-角色关联表
    ↓ N:1
sys_role            角色表（SYSTEM_ADMIN / ROLE_ADMIN / CATEGORY_ADMIN）

fb_category_admin   类别-管理员关联表（控制 CATEGORY_ADMIN 可见的反馈类别）
    ↓
fb_category         反馈类别表
```

### 3.2 表关系说明

- 一个用户可以拥有多个角色（如同时是 ROLE_ADMIN 和 CATEGORY_ADMIN）
- 一个角色可以分配给多个用户
- CATEGORY_ADMIN 的数据权限通过 `fb_category_admin` 表控制：
  - 该表记录"哪个管理员负责哪个反馈类别"
  - 查询反馈列表时，CATEGORY_ADMIN 只能看到被分配类别下的反馈

## 四、后端鉴权方案

### 4.1 技术实现

采用**自定义注解 + 拦截器**方案（不引入 Spring Security）：

```
请求 → JwtInterceptor（认证 + 加载角色） → RoleInterceptor（角色校验） → Controller
```

1. **JwtInterceptor**（已有，需增强）：
   - 验证 JWT token 有效性
   - 解析 userId、username、userType
   - **新增**：查询 `sys_user_role` + `sys_role` 表，加载用户角色列表到 UserContext

2. **RoleInterceptor**（新建）：
   - 读取 Controller 类/方法上的 `@RequiresRole` 注解
   - 校验当前用户的 userType 或 roles 是否满足要求
   - 不满足返回 `403 权限不足`

### 4.2 @RequiresRole 注解使用方式

```java
// 仅学生可访问
@RequiresRole(userType = "student")
public class StudentFeedbackController { ... }

// 需要特定角色之一
@RequiresRole({"SYSTEM_ADMIN", "ROLE_ADMIN"})
public class AdminCategoryController { ... }

// 方法级别覆盖类级别
@RequiresRole({"SYSTEM_ADMIN", "ROLE_ADMIN", "CATEGORY_ADMIN"})
public class AdminFeedbackController {

    @RequiresRole({"SYSTEM_ADMIN"})  // 更严格的方法级注解
    public Result<?> deleteAll() { ... }
}
```

### 4.3 接口权限配置清单

| 接口路径 | 权限要求 |
|---------|---------|
| `POST /api/auth/login` | 免认证 |
| `GET /api/auth/me` | 仅需认证（任何已登录用户） |
| `POST /api/auth/logout` | 仅需认证 |
| `GET/POST /api/student/**` | userType = student |
| `GET/POST /api/admin/feedback/**` | SYSTEM_ADMIN / ROLE_ADMIN / CATEGORY_ADMIN |
| `GET/POST/PUT/DELETE /api/admin/category/**` | SYSTEM_ADMIN / ROLE_ADMIN |
| `GET/POST/PUT/DELETE /api/admin/quick-reply/**` | SYSTEM_ADMIN / ROLE_ADMIN / CATEGORY_ADMIN |
| `GET /api/admin/statistics/**` | SYSTEM_ADMIN / ROLE_ADMIN |
| `GET/POST/PUT/DELETE /api/system/user/**` | SYSTEM_ADMIN |
| `GET /api/system/role/**` | SYSTEM_ADMIN |
| `GET/PUT /api/system/category-permission/**` | SYSTEM_ADMIN / ROLE_ADMIN |
| `GET/POST/PUT/DELETE /api/base/**` | SYSTEM_ADMIN |
| `POST /api/upload/file` | 仅需认证（任何已登录用户） |
| `/api/files/**` | 免认证（静态资源） |

### 4.4 CATEGORY_ADMIN 数据过滤

CATEGORY_ADMIN 在访问反馈相关接口时，需要在 Service 层进行数据过滤：

- **反馈列表查询**：自动添加 `WHERE category_id IN (被分配的类别ID列表)` 条件
- **反馈详情查看**：校验该反馈的 categoryId 是否在允许列表中
- **反馈状态统计**：仅统计被分配类别下的反馈数量

SYSTEM_ADMIN 和 ROLE_ADMIN 不受此限制，可查看所有类别的反馈。

## 五、前端权限方案

### 5.1 路由守卫

在 `router.beforeEach` 中增加角色检查：

- 路由 meta 配置 `requiredUserType` 或 `requiredRoles`
- 用户访问无权限的路由时，自动重定向到有权限的默认页面
- student → `/student/feedback`
- admin/teacher → `/admin/feedback`（需有对应角色）

### 5.2 导航栏动态显示

顶部导航 Tab 根据用户角色动态过滤：

| Tab | 可见条件 |
|-----|---------|
| 我要反馈 | userType = student |
| 反馈跟踪 | userType = student |
| 反馈查看 | SYSTEM_ADMIN / ROLE_ADMIN / CATEGORY_ADMIN |
| 反馈统计 | SYSTEM_ADMIN / ROLE_ADMIN |
| 反馈管理 | SYSTEM_ADMIN / ROLE_ADMIN |
| 系统管理 | SYSTEM_ADMIN / ROLE_ADMIN（子页面各自控制） |

系统管理子页面：

| 子Tab | 可见条件 |
|-------|---------|
| 用户管理 | SYSTEM_ADMIN |
| 学期管理 | SYSTEM_ADMIN |
| 年级管理 | SYSTEM_ADMIN |
| 班级管理 | SYSTEM_ADMIN |
| 师生关系 | SYSTEM_ADMIN |
| 角色权限 | SYSTEM_ADMIN |
| 类别权限 | SYSTEM_ADMIN / ROLE_ADMIN |
| 快捷回复 | SYSTEM_ADMIN / ROLE_ADMIN / CATEGORY_ADMIN |

### 5.3 登录跳转逻辑

```
登录成功
  ├── userType = student → /student/feedback
  ├── 拥有任意 admin 角色 → /admin/feedback
  └── teacher 无角色 → 提示"暂无可用功能，请联系管理员分配权限"
```

## 六、权限管理功能

### 6.1 角色权限设置页面（/system/role）

**功能**：管理员可以给用户分配或撤销角色。

**页面布局**：
- 左侧：角色卡片列表（系统管理员、角色管理员、类别管理员）
- 右侧：选中角色下的用户列表
- 操作：搜索用户 → 添加到角色 / 从角色移除

**接口**：
- `GET /api/system/role/list` — 获取角色列表
- `GET /api/system/role/users?roleId=xxx` — 获取角色下的用户
- `POST /api/system/role/assign` — 分配角色
- `DELETE /api/system/role/revoke` — 撤销角色

### 6.2 类别权限配置页面（/system/category-permission）

**功能**：为 CATEGORY_ADMIN 分配负责的反馈类别。

**页面布局**：
- 表格展示所有反馈类别
- 每行显示：类别名称 | 负责管理员（Tag标签） | 编辑按钮
- 点击编辑弹出对话框，搜索并多选管理员

**接口**：
- `GET /api/system/category-permission/list` — 获取类别权限列表
- `PUT /api/system/category-permission/{categoryId}` — 更新类别的负责管理员
